# 与已有研究的对比分析

> 以下内容应作为论文第1.2节（研究动机）的补充，或作为独立的"相关工作"章节插入第2节之前。

## 相关工作与本文的区分

AI辅助编程的研究在过去三年经历了爆发式增长。本节将本文的工作置于这个快速演进的文献脉络中，明确说明本文与已有研究的区别——不是什么"填补空白"，而是"在已有地基上砌了一面特定的墙"。

### 1. 代码生成能力评估

Chen et al. (2021) 的 Codex 论文[2]通过 HumanEval 基准首次系统性地评估了 LLM 的代码生成能力，报告了 28.8% 的 pass@1。Austin et al. (2021) 的 MBPP 基准[3]将这一数字推至 59.6%（few-shot）。这两篇奠定了"用标准化编程问题评估 LLM"的范式。

**本文与它们的区别**：HumanEval 和 MBPP 的任务都是独立的函数级编程题，平均长度约 10-30 行。本文的案例是模块级的 C++ 工程任务（环形缓冲区 140 行、HTTP 客户端 200 行、日志系统 300+ 行），涉及跨文件依赖、构建配置和真实并发场景。这个粒度差异不是量的区别——模块级任务引入了函数级评测不会触及的问题：跨文件的接口一致性、构建系统的配置正确性、以及多轮对话中的上下文保持。

### 2. 开发者生产力影响

Peng et al. (2023) 的 RCT[9]报告了 GitHub Copilot 使任务完成速度提升 55.8%——这是一个受到广泛引用的数据。但该研究采用的是单一任务（JavaScript HTTP 服务器）和单语言设计。Mastropaolo et al. (2023) 的鲁棒性研究[7]则揭示了语义等价描述改变后代码生成正确率会显著下降。

**本文与它们的区别**：首先，本文聚焦 C++——一种具有独特复杂性（模板元编程、内存模型、RAII）的语言，这在 AI 辅助编程文献中受到的关注远少于 Python 和 JavaScript。其次，本文不是报告一个单一的效率提升数字，而是区分了 Claude Code 在"常规场景"（案例一、三）和"高风险场景"（案例二协程、案例四模板元、案例五底层寄存器）之间的能力落差——这种区分比一个笼统的"效率提升 X%"更有实践指导意义。

### 3. 工具使用与实践研究

Rasnayaka et al. (2024)[8]在 214 名学生的软工项目中观察 LLM 的使用，发现 AI 工具主要用于基础代码结构和语法帮助。Santos et al. (2026) 分析了 328 个 CLAUDE.md 配置文件来理解 Claude Code 的使用模式。

**本文与它们的区别**：Rasnayaka 的研究对象是学生，本文的操作者是具有 3 年以上 C++ 开发经验的工程师。Santos 的研究是配置文件的大规模文本分析，而本文是工具的深度使用记录——前者告诉你"人们怎么配置 Claude Code"，后者告诉你"配置完了它能做什么、会出什么错"。

### 4. 本文的独特贡献

综合以上对比，本文的贡献不在于"第一个研究 Claude Code 在 C++ 上的应用"（这个时间窗口已经过去了），而在于提供了一份**带具体 bug 记录、修复过程、多轮对话日志和失败案例的技术实践文档**，其独特之处在于：

1. **模块级粒度**：不是在函数级 benchmark 上打分，而是在真实 C++ 模块（含头文件组织、构建配置、并发测试）上记录表现
2. **失败透明**：每个案例都记录了 Claude Code 的具体缺陷和修复过程，而非仅报告成功
3. **能力边界映射**：区分了"能做"、"勉强能做"、"做不了"三个区域，为从业者提供了实用的工具选择参考
4. **可复现性**：提供了完整的代码仓库、对话日志和实验协议（见 `claude-code-cpp-study/`）

这些贡献的学术价值不在于"创新性"或"理论突破"，而在于为 AI 辅助软件工程这个快速演进的领域提供了一份有据可查的实践快照——类似于医学中的病例报告（case report），其价值在于细节的真实性和可复现性，而非统计显著性。
