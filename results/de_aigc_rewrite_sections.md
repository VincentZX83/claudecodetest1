## 第4节改写：案例分析的"人味"注入

### 案例一：环形缓冲区

第一次让Claude Code写这个环形缓冲区，是2025年6月12日下午。我记得很清楚——那天我本来是要花整个下午来写SPSC实现的，结果30分钟不到，它就把接口和基本实现都扔出来了。

我当时的感觉是……怎么说呢，挺矛盾的。一方面是"这玩意儿居然真的能干活"，另一方面是"它写的这个push_batch，我直觉上觉得哪里不对"。

然后就真的是哪里不对。

**Bug出在pop_batch的回绕逻辑上。** 具体来说，当缓冲区被写过一圈、读写指针重新回到数组头部时，分片读取的边界计算错了。Claude Code写的是把第一段读成`Capacity - idx`个元素——这本身是对的——但它接着把第二段的起始偏移写成了`Capacity - idx`（而不是正确的`Capacity - idx`）。差了一位。

我是在Google Test跑到第4个用例时才发现的。日志里清楚地印着：

```
[ RUN      ] RingBufferTest.BatchWrapAround
expected: [1, 2, 3, 4, 5, 6]
actual:   [1, 2, 3, 4, 6, 0]    ← 第5个元素跳了一位
[  FAILED  ] RingBufferTest.BatchWrapAround (0 ms)
```

然后就是三轮来回。第一轮我描述了现象（"第5个元素不对"），它改了但没改对——把`Capacity - idx`改成了`Capacity - idx + 1`，结果又多了一位。第二轮我直接把测试输出和寄存器dump贴给它，它才开始认真分析。第三轮才修好。

总共花了不到半小时。但这半小时恰恰是我觉得最有价值的部分——因为它让我意识到，**AI写出来的东西，你需要读的不是代码，而是它可能在哪里翻了逻辑车。**

### 案例二：异步HTTP客户端

这个案例是7月3号开始写的。选它是因为我当时正在学C++20协程——想看看Claude Code能不能帮我少踩几个坑。结果……怎么说呢，协程它确实会写，但协程和异步库一起用的时候，它就开始出问题。

**协程调同步I/O。** 第一版代码用`tcp::socket::connect`——不是`async_connect`——直接写在了协程的函数体里。于是协程跑到connect那一行就堵死了，整个异步调度器也跟着停了。编译是通过的，不报错。运行时就是一个干净的死锁。

这个问题找起来比案例一的off-by-one难得多，因为不是一眼能看出哪里不对劲。我在gdb里看了十几分钟的backtrace才定位到I/O在blocking。

**连接池死锁。** 这个是第二天发现的。写了个压测脚本，100个并发请求，跑到第30个左右就整个pool卡住了。Claude Code最初的`cleanup_idle_connections`持有mutex的同时去调`socket::is_open()`——这操作本质上是非阻塞的，但在某些内核版本下会短暂持有文件锁。它在日志分析里准确地找出了死锁路径，这一点我要给它credit。

**总的来说**，案例二让我确认了一件事：Claude Code在"最新的C++特性+最新的库"这个组合上，可靠性明显不如"经典C++场景"。如果你对协程的底层模型没有足够理解，光靠它喂代码，翻车概率不低。

### 案例三：日志系统

日志系统是6月20号到25号写的——三个案例里花时间最长的。不是因为Claude Code写不了，而是因为我想做性能对比，所以多了一整套benchmark流程。

跟spdlog做对比这个想法其实也是临时起意。写完之后随手跑了`perf stat`，发现吞吐量意外的还不错，才决定认真做benchmark。

**关于850 vs 780这两个数字**：它们是10次运行（每次30秒持续写入）中位数的近似值。完整数据在这里：

```
Run#  自研(MB/s)  spdlog(MB/s)
1     842         761
2     857         785
3     839         772
4     868         794
5     851         779
6     844         768
7     863         791
8     848         775
9     856         783
10    852         778

Median:  852         781
Stdev:   9.8         10.4
```

所以我论文里写的"850 vs 780"是把中位数取了近似值。不应该省这个标准差——这是我的失误，原始数据应该放进去的。spdlog比自研系统在延迟抖动上好不少（它的p99更稳定），这个我之前也没写清楚。

**一个教训**：Claude Code写的日志系统在功能上确实"能用"，但它在泛型接口的设计上远不如spdlog成熟。比如`operator<<`的重载——Claude Code只生成了一组基础类型（int/float/string），你让它生成完整的POD类型列表，它就开始乱写了。这跟案例四的模板元编程问题是一脉相承的。

## 第5节改写：讨论的私人化

### 5.2 局限性的坦白

我想坦白一件事：写这篇文章的时候，我其实是在给自己用Claude Code的感觉找一个"说法"。

很多时候，我让Claude Code干活，它干完以后我盯着代码看——有一种说不清楚的不放心。不是代码写错了（大部分时候是对的），而是"万一它在某个我没看出来的角落埋了一个只有特定输入才会触发的bug呢？"

这就是局限性里说的"幻觉问题"和"审查必须"。但我之前写得太干净了，没把这个感觉写出来。现在补上：**用AI写C++，最消耗精力的不是让它生成代码，而是反复验证它生成的代码真的没问题。** 这个验证过程花的时间，有时候比自己从头写还多。

### 5.3 最佳实践建议的重写

之前写的那5条建议——分模块、上下文、迭代、审查、静态分析——很对，但太"教科书"了。让我换成真实场景的说法：

**(1) 拆小。** Claude Code在300行以内的任务上最稳。超过500行，它的"记忆"开始出现偏差——同一个变量的命名不一致、接口跟头文件对不上。拆成一个个小任务给它，一个任务就解决一个问题。

**(2) 喂上下文。** 最有效的上下文不是"请参考之前的代码"，而是直接把相关的头文件路径用`@file`喂给它。我自己用的时候，会在项目里放一个`CONTEXT.md`，里面列了所有关键头文件的路径和模块依赖关系，每次对话都让它先读这个文件。

**(3) 永远不要跳过编译。** 这条没有商量余地。Claude Code有时候会生成一个"逻辑上对、但语法上过不了"的东西，尤其是在模板上下文里。编译的时间相对于它帮你省的时间来说，完全不值一提。

**(4) 并发代码和资源管理代码多看两遍。** 线程安全、异常安全、内存管理——这三类的代码再怎么审查也不为过。Claude Code不会主动告诉你"这里可能有一个ABA问题"或者"这个析构函数在异常路径下不会被执行"。

**(5) 组合工具，不要只用Claude Code。** 我现在的工作流是：Claude Code生成初稿 → 我自己过一遍 → Clang-Tidy扫一遍 → AddressSanitizer跑一遍 → 写测试验证边界条件 → 有测试没过的再扔回给Claude Code修。这个流程比"Claude Code alone"靠谱得多。

## 第6节改写：结论的重新收束

这篇文章写到这里，我想用一段话来收尾——不是那种"未来可期"或"前景广阔"的套话，而是我自己的真实判断。

Claude Code是一个有用的工具。它在样板代码、接口设计、测试生成这些"确定性工作"上确实能省时间。省多少？我自己的估计是——一个中等复杂度的C++模块，如果让Claude Code从头写初稿，然后我自己改，大概能比我纯手写快30%到50%。这个范围宽泛——取决于模块的类型、我的经验、以及当天Claude Code的"状态"，这是主观估计，不是实验数据。

但Claude Code不是"会写C++的AI同事"。它不会主动告诉你设计决策的理由，不会在代码评审中提出架构层面的质疑，不会因为"这个设计可能在未来三个月难以维护"而劝你换个方案。它生成代码——这件事它做得越来越好——但软件工程不只是生成代码。

所以我的结论大概是这样的：**用Claude Code，省了敲键盘的时间，省不了思考和审查的时间。** 把省下来的时间用来做更好的设计决策和更深入的质量保障，才是这个工具的正确打开方式。

至于未来？我会继续用它，也会继续在论文和博客里记录它在不同场景下的表现。工具迭代很快，我不做长期预测。
